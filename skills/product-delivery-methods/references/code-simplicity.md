# Code Simplicity

来源：`DietrichGebert/ponytail`（MIT，upstream commit `45f7d2f83fb430a65fd512a98ad7b14d79e06636`）。

## 适用阶段

- `Dev`：写实现前，用它限制无必要抽象、依赖、样板代码和文件膨胀。
- `Review`：审查 diff 时，额外找可删除、可合并、可用标准库或平台能力替代的代码。
- `Smoke / QA`：保留最小可运行检查，不能用“少写代码”跳过验证证据。

## 编码阶梯

写代码前按顺序停在第一个成立的选项：

1. 不需要实现的，直接跳过并说明原因。
2. 标准库已经覆盖的，用标准库。
3. 浏览器、数据库、框架或平台原生能力已经覆盖的，用原生能力。
4. 项目已安装依赖可以稳定覆盖的，复用既有依赖。
5. 一行能正确表达的，不拆成多层结构。
6. 以上都不成立时，只写满足当前验收标准的最小实现。

## 硬边界

- 不为单个实现新增 interface、factory、adapter 或抽象层。
- 不为“以后可能需要”新增配置、扩展点、脚手架或目录。
- 不新增依赖来替代几行清晰代码，除非它已经是项目标准依赖或解决真实高风险问题。
- 优先删除、内联、合并和复用，避免为了“架构感”扩大 diff。
- 同等简洁时，选择边界更正确的标准库/平台能力，不选脆弱算法。

## 不能省的部分

- 信任边界输入校验。
- 防止数据丢失的错误处理。
- 安全、权限、审计、资金、隐私相关控制。
- 可访问性基础要求。
- 真实硬件、时间、传感器、第三方系统需要的校准和容错。
- 用户明确要求的完整能力。

## 验证要求

- 非平凡逻辑必须留下一个最小可运行检查：一个断言、自检脚本、小测试或冒烟步骤均可。
- 只有纯映射、纯样式微调、明显一行替换这类低风险改动，才允许不新增测试。
- 如果为了交付速度有意留下简化实现，在代码附近写 `ponytail:` 注释，说明上限和升级触发条件。

示例：

```python
# ponytail: O(n) scan is fine below 10k rows; add indexed lookup when profiling shows this path hot.
```

## Review 输出口径

复杂度审查使用以下标签：

- `delete:` 死代码、未用功能、无必要弹性。
- `stdlib:` 手写了标准库已有能力。
- `native:` 依赖或代码替代了平台原生能力。
- `yagni:` 单实现抽象、无人配置的配置项、只有一个调用方的层。
- `shrink:` 同样逻辑可以更少代码表达。

输出优先写成可执行建议：

```text
path/to/file.ts:L42: yagni: FooFactory 只有一个实现。内联到调用方，等第二个实现出现再抽象。
```

## 与 product-delivery 的关系

`ponytail` 在这里是方法增强，不是主流程替代。它只影响“怎么写得更少、更稳”，不允许跳过 `Requirement -> Design -> Dev -> Review -> Smoke -> QA -> Release` 的证据链。
