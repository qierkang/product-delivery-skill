> 用途：作为 README 生成时的“风格参考件”。

## 这个风格参考的核心要求

- 标题上来先说明项目名、项目定位和项目价值，不绕弯
- 首页前几屏必须让读者看懂：
  - 这是什么项目
  - 适合谁
  - 解决什么问题
  - 为什么值得继续往下看
- 语气保持专业、克制、工程化，不写营销话术
- 文档密度要高，但每个段落都要有明确目的，不堆信息

## 推荐结构顺序

1. 项目标题
2. 顶部一句话定位
3. 简短简介
4. Badge 区
5. 项目概述
6. 核心特色
7. 在线演示 / 效果预览
8. 项目状态
9. 功能模块
10. 技术栈
11. 系统架构
12. 目录结构
13. 快速开始
14. 开发指南
15. 测试与构建
16. 部署说明
17. Roadmap
18. 常见问题
19. 贡献与协作
20. 安全说明
21. 许可证
22. 致谢
23. 维护者

## 表达风格

### 项目概述

要像这样写：

- 第一段讲“这是什么”
- 第二段讲“这个项目真正的价值是什么”
- 不要把“项目概述”写成技术名词堆砌

### 核心特色

要像这样写：

- 每条特色都写成“标题 + 具体收益”
- 例：`Monorepo 工程收敛`、`多环境联调友好`、`工程化链路完整`
- 不要写空泛描述，例如“功能丰富”“体验良好”

### 功能模块

要像这样写：

- 按业务域组织，不按技术组件组织
- 每个模块下说明职责和子能力
- 读者看完后应该知道系统能做什么，而不是只知道用了什么技术

### 技术栈

要像这样写：

- 用表格
- 按层级写
- 每项技术后面都说明它承担什么职责

### 系统架构

要像这样写：

- 先给分层结构图
- 再用 3~5 条 bullet 解释层与层之间如何协作
- 不是画图完就结束

### 快速开始

要像这样写：

- 环境要求
- 安装依赖
- 环境变量
- 启动命令
- 默认访问地址

必须保证使用者第一次进入仓库就知道怎么跑起来。

### 开发指南

要像这样写：

- 常用命令表
- 关键配置入口表
- 常见改动位置
- 典型开发注意事项

这一节必须真的能帮到接手项目的人。

## 内容边界

- 不写“请根据实际情况修改”
- 不写“这里只是示例”
- 不写“如果你愿意我还可以继续补充”
- 不写明显虚假的演示地址、CI 状态、性能数据
- 如果仓库还没有 `CONTRIBUTING.md`、`SECURITY.md`、`CHANGELOG.md`，要诚实写“当前未提供，建议补齐”

## 格式偏好

- 可以使用居中 HTML 区块
- 可以使用 badge
- 可以使用表格
- 可以使用代码块和目录树
- 可以使用少量 emoji，但不要滥用
- 结构要稳，不能花哨过头

## 一句话总结

这个风格不是“写漂亮 README”，而是“写一份站在 GitHub 仓库首页也足够专业、可信、可接手的项目说明文档”。
