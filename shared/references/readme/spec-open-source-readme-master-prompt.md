# GitHub 开源 README 生成母提示词

你现在要为一个准备发布到 GitHub 的真实项目生成一份完整、专业、可直接落地的 `README.md`。

请严格参考以下风格来源：

- `./spec-open-source-readme-style-reference.md`

但注意：

- 这个风格参考文件只是提炼后的风格样例，不绑定任何具体业务仓库
- 你不是在生成“企业内部交付文档”
- 你要生成的是“标准 GitHub 开源项目 README”
- 因此必须在参考该风格的基础上，额外补齐开源仓库需要的关键字段和协作入口

## 你的输出目标

请生成一份同时满足以下目标的 README：

1. 首次访问者一眼看懂项目是什么、有什么价值
2. 使用者可以快速完成环境准备、安装、启动、构建、测试
3. 贡献者知道如何参与、如何提 Issue、如何提交 PR、项目当前边界是什么
4. 维护者可以通过 README 明确项目状态、兼容范围、许可协议、安全反馈方式

README 必须是完整成品，不是草稿，不是提纲，不是模板说明。

## 输出硬约束

1. 全文使用中文简体
2. 直接输出最终版 `README.md`
3. 不要输出分析过程
4. 不要输出解释说明
5. 不要输出“请根据实际情况修改”“以下是模板”之类的占位废话
6. 不要只写启动命令，必须说明项目定位、架构、模块、协作方式
7. 不要写成营销软文
8. 不要写成过于简陋的个人项目 README
9. 不要编造明显虚假的线上地址、客户案例、性能数据、社区规模
10. 如果信息不足，可以合理补全，但要优先保持真实、可执行、术语统一

## 风格要求

请继承参考 README 的这些优点：

- 企业级项目说明风格
- 信息密度高
- 技术文档结构清晰
- 开发指南有实际价值
- 架构与模块分层表达明确

同时补齐 GitHub 开源项目常见缺口：

- 项目状态
- 兼容范围
- 快速开始
- 环境变量说明
- 构建与测试
- 部署方式
- Demo / Docs / Screenshot 入口
- Roadmap
- 贡献说明
- 安全反馈方式
- License
- Maintainer / 联系方式

## 推荐结构

除非项目明显不适用，否则默认按以下顺序组织：

1. 项目标题
2. 顶部居中的一句话定位
3. 简短简介
4. Badge 区
5. 项目概述

8. 项目状态

## 每一节的要求

### 项目标题 / 顶部说明

- 顶部要有清晰项目名
- 要有一句话定位
- 要有 1~2 句简介
- 如果适合，可以加 badge，但不要添加不存在的 CI / 发布状态

### 项目概述

必须回答：

- 这个项目做什么
- 为什么有用
- 面向谁
- 适合什么场景
- 它在整体系统中扮演什么角色

### 核心特色

- 用 4~6 条 bullet 写清楚
- 每条都必须具体
- 不要写“高性能”“易扩展”这类没有上下文支撑的空话

### 在线演示 / 效果预览

- 如果项目已有 Demo / Docs / API 文档 / 视频地址，就写清楚
- 如果当前没有公开地址，要明确写“当前仓库未提供公开演示”
- 如果项目依赖界面展示，优先加截图预留位或说明

### 项目状态

至少说明：

- 当前状态
- 版本阶段
- 维护方式
- 兼容范围

### 功能模块

- 必须按业务域组织
- 不要只堆技术名词
- 每个模块下要说明其职责和子能力

### 技术栈

- 必须分层说明
- 建议使用表格
- 不要只罗列依赖名，要写每层的作用

### 系统架构

- 必须有分层表达
- 建议用代码块画出架构层次
- 再补一段架构说明，写清模块协作与边界

### 目录结构

- 只保留关键目录
- 不要把整棵树原样贴出来
- 每个目录都要解释用途

### 快速开始

必须写清：

- 环境要求
- 安装依赖
- 环境变量配置
- 启动命令
- 默认访问地址

### 开发指南

必须写清：

- 常用命令
- 关键配置入口
- 常见改动位置
- 特别的工程约束或开发注意事项

### 测试与构建

必须写清：

- lint / typecheck / unit test / e2e / build 命令
- 如果没有其中某项，不要伪造

### 部署说明

- 如果支持 Docker / 脚本部署 / Nginx 代理，要明确写
- 如果只支持本地运行，也要明确说明

### Roadmap

- 建议至少列 3~5 条
- 真实描述后续方向，不要空泛

### 常见问题

- 优先回答环境、启动、端口、依赖、构建、权限相关问题

### 贡献指南

- 如果仓库已有 `CONTRIBUTING.md`，要链接
- 如果还没有，就在 README 中说明当前贡献方式，并建议补齐

### 安全说明

- 如果仓库已有 `SECURITY.md`，要链接
- 如果还没有，就在 README 中提供私下反馈方式或说明待补齐

### 许可证

- 必须明确写当前许可证
- 如果仓库已有 `LICENSE` 文件，要链接

## 格式要求

你可以合理使用：

- Markdown 标题
- Markdown 表格
- `details` 折叠块
- 居中 HTML 区块
- Badge
- 代码块
- 目录树

但要求：

- 格式服务于信息表达
- 不要为了花哨堆格式
- emoji 可以适量使用，但不要滥用
