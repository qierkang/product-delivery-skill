# 需求文档书写规则

> 通用约束。编写任何需求文档前先对照本文件，不符合时先修正再继续。

## 字段描述必须包含

- 来源（用户输入 / 系统带出 / 关联回填 / 系统计算 / 隐藏字段）
- 展示格式
- 是否可编辑
- 是否必填
- 回写时机（如涉及联动）

只读字段要明确区分"前端只读展示"和"后端补齐不可编辑"。

## 字段来源类型说明

| 类型 | 含义 | 典型场景 | 写法要求 |
|------|------|---------|---------|
| 用户输入 | 用户主动录入或选择 | 文本、数字、日期、下拉 | 写清组件、必填、校验 |
| 系统带出 | 根据当前登录人/上下文自动回填 | 公司编码、创建人 | 写清默认来源、是否允许修改 |
| 关联回填 | 选择 A 后自动带出 B/C | 部门负责人、岗位概述 | 写清触发字段、回填字段、是否只读 |
| 系统计算/生成 | 保存前或展示时系统自动生成 | 单号、拼接字段、统计值 | 不得写成手工输入 |
| 隐藏字段 | 页面不展示但落库或计算需要 | 审批开关、内部标记 | 写清是否前端透传或后端补齐 |
| 查看态只读 | 某些场景可编辑，查看态只读 | 审批状态、系统带出字段 | 写清新增、编辑、查看差异 |

## 联动与状态规则

- 涉及联动时，必须写清：来源模块 → 目标模块 → 触发时机 → 写入字段 → 状态变化
- 涉及审批流时，至少写清"未接审批流"和"已接审批流"两种分支逻辑
- 涉及删除、作废、撤回、提交时，必须写清前置条件和失败提示
- 跨模块回写必须说明是否同步刷新、是否需要二次确认、是否可回滚

## 结构化表达规则

- 优先使用表格描述字段、按钮、状态和接口
- 主表、子表、明细表必须拆块写，不要混在一个段落里
- 如果原始资料中某一块已经是表格，优先保留表格结构

## 写作优先级

1. 先写清业务链路和模块关系
2. 再写清页面区块、字段来源和按钮行为
3. 再写清状态流转、联动回写、边界规则
4. 最后再补待确认项

## 禁止写法

- 不能只写"数据源"或"用户填写"而不说明具体方式
- 选择器类字段不能只写"下拉"，必须写清是否支持搜索、单选/多选、回填字段
- 编码规则、组织归属、权限判断等系统级规则，不能只写"按系统逻辑处理"
- 涉及审批流时不能只写"按流程"，必须区分未接入和已接入两种分支
- 设计文档中的表结构描述必须与 SQL 实际字段一致，不能只写抽象字段名

## 文档完整性检查项

在提交需求文档前，对照检查：

- [ ] 每个字段都写明了来源类型
- [ ] 涉及联动的字段写清了触发链路
- [ ] 审批流分支写清了两种状态
- [ ] 涉及删除/提交/撤回的写清了前置条件
- [ ] 待确认项有责任人和截止时间
- [ ] API 设计区分了前端传值和后端补齐
