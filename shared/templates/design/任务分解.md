## 1. 分解原则

1. 先文档后开发，先主链路后边界场景，先公共依赖确认后编码
2. 每项任务粒度控制在 30-120 分钟可交付
3. 每项任务必须有"输入、输出、验收标准、证据"
4. 失败任务必须有回流动作和责任人
5. 同一任务连续失败 3 次必须升级拆解
6. 任务必须覆盖需求、设计、开发、自测、审查、冒烟、QA、UI、产品验收、发布全链路

## 2. 阶段任务清单

| 阶段 | 任务ID | 任务内容 | 负责人 | 输入 | 输出 | 验收标准 | 证据要求 | 回流动作 |
|---|---|---|---|---|---|---|---|---|

### 2.1 产品与需求阶段

- 目标：明确范围、目标、边界、依赖、风险。
- 输出：`需求文档.md`、`00-需求总览.md`、`manifest.json`

### 2.2 架构与 UI 阶段

- 目标：完成技术方案、UI 规范、服务端任务、页面接口对照、任务切片。
- 输出：`技术方案.md`、`UI交互设计规范.md`、`服务端开发任务.md`、`任务分解.md`

### 2.3 开发阶段

- 目标：完成实现、联调、自测和证据回填。
- 输出：`实现控制总表.md`、`页面接口验收总表.md`、`开发放行报告.md`、`覆盖率报告.md`、`前端关键流程覆盖清单.md`

### 2.4 审查与冒烟阶段

- 目标：完成代码审查和主链路冒烟复核。
- 输出：`代码审查报告.md`、`冒烟测试脚本.sh`、`冒烟测试报告.md`

### 2.5 QA 与验收阶段

- 目标：完成测试用例、QA 验收、UI 验收、产品验收。
- 输出：`测试用例.md`、`QA验收报告.md`、`UI验收报告.md`、`产品验收报告.md`

### 2.6 发布阶段

- 目标：完成发布前检查、发布执行、发布后验证和记录回写。
- 输出：`发布记录.md`、`stage-status.json`

## 3. Gate 检查点

| Gate | 触发时机 | 必需输入 | 必需输出 | 通过条件 | 不通过动作 |
|---|---|---|---|---|---|
| Requirement | 需求文档完成后 | 需求文档、技术/UI 基线 | 需求总览 | 范围清晰、边界明确 | 回 PM 重拆 |
| Design | 方案阶段完成后 | 技术方案、UI 规范 | 任务分解 | 任务可执行、证据要求明确 | 回方案重拆 |
| Dev | 编码完成后 | 代码变更、本地环境 | 开发放行报告、覆盖率报告、前端关键流程覆盖清单 | 0 阻塞项 | 回开发重测 |
| Review | Dev Gate 通过后 | 代码 diff、编译结果 | 代码审查报告 | 0 P0/阻塞缺陷 | 回开发修复 |
| Smoke | Review 通过后 | 运行环境、冒烟脚本 | 冒烟测试报告 | 主链路 100% 通过 | 回开发/配置修复 |
| QA | Smoke 通过后 | 测试用例、接口、冒烟结果 | QA验收报告 | 关键缺陷清零 | 回开发 |
| UI Acceptance | QA 通过后 | 页面、交互规范 | UI验收报告 | 视觉和交互无阻塞 | 回前端/UI |
| Product Acceptance | UI 验收通过后 | 业务基线、页面结果 | 产品验收报告 | 业务符合且边界清晰 | 回开发/PM |
| Release | 全量验收通过后 | 全量报告 | 发布记录 | 可回滚，发布后验证通过 | 暂缓发布 |

## 4. 角色分工

| 角色 | 主要职责 | 不负责事项 |
|---|---|---|
| 总控（Claude） | 总控调度、阶段推进、卡点管理、文档产出 | 不直接跳过需求和方案写代码 |
| 执行（Codex） | 服务端方案、任务推进、编码、公共依赖确认 | 不直接定稿 UI、不擅自改需求边界 |
| 前端线程 | UI 规范、三端差异、交互约束、前端实现 | 不定义服务端接口语义 |
| QA | 功能验证与缺陷回流 | 不替代开发自测 |
| UI / 产品 | 页面、交互、业务验收与缺陷回流 | 不替代 QA 功能验证 |

## 5. Gate 前置条件

- 无需求基线，不进入技术方案
- 无技术方案，不进入编码
- 无 UI 规范，不进入前端/小程序/App 页面开发
- 无开发放行报告，不进入 review
- 无覆盖率报告与前端关键流程覆盖清单，不进入 review
- 无代码审查与冒烟结果，不宣称可验收
- 无 QA 验证结果，不宣称可进入 UI / 产品验收
- 无 UI 验收结果，不宣称可进入产品验收
- 默认结论不是 `READY`，而是 `NEEDS_WORK`

## 7. Dev↔QA 回环策略

- QA 提出问题后，先回写问题清单，再回到开发阶段修复
- 修复完成后，必须补新的验证证据
- 不允许只口头回复"已修复"，必须落报告
- 每个缺陷必须有：发现时机 → 修复说明 → 回归证据
