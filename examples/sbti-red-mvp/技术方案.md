## 1. 架构设计

- 前台应用：`apps/web`，技术栈保持 `React + Vite`，聚焦 H5/Web 体验。
- 后台应用：新增 `apps/manage`，技术栈同样保持 `React + Vite`，用于运营查看。
- API 服务：`apps/api`，保持 `Node.js + Express`，统一承载测试、报告、邮件、海报、后台接口。
- 数据层：`MySQL 8`（业务数据）+ `Redis 7`（在线心跳/短缓存）。
- 文件层：本地文件目录 `storage/`，用于海报、生成图片、上传文件；后续可抽象到 OSS。
- 部署：`Docker Compose`，公共依赖与前后台应用分层部署。

## 4. 关键流程

### 4.1 免费主流程

1. 用户进入首页，请求 `/api/session/bootstrap`。
2. 后端返回：
   - `session_id`
   - 默认随机昵称
   - 首屏配置
3. 用户可填邮箱或跳过；如果填写，则调用 `/api/leads/subscribe`。
4. 用户完成测试后调用 `/api/tests/submit`。
5. 后端同步完成：
   - 计算人格编码与维度分
   - 写入 `quiz_submissions`
   - 组装免费报告 JSON
   - 生成 `report_token`
   - 计算今日第 N 位
6. 前端跳转结果页 `/report/:reportToken`，请求 `/api/reports/:reportToken`。
7. 结果页异步触发海报生成 `/api/reports/:reportToken/poster`。

### 4.2 邮件流程

1. 用户结果页留邮箱或首次已留邮箱。
2. 前端调用 `/api/emails/report/send`。
3. 服务端创建 `email_jobs` 记录并发送：
   - 免费版发送“基础报告 + 访问链接”
   - 已解锁后发送“详细报告入口”

### 4.3 详细报告解锁流程（预留）

1. 用户点击“解锁详细解答”。
2. 前端进入 `/report/:reportToken/detail`。
3. 若 `unlock_status=locked`，展示预览 + 支付方式。
4. 支付接口未配置时返回“即将开放”；配置后走 H5 下单与回调。

### 4.4 复访与缓存流程

1. 前端把 `report_token` 存在 `localStorage`。
2. 用户再次打开站点，若有有效 `report_token`，直接优先进入报告页。
3. 用户点击“重新测试”后，原报告保留历史记录，新结果覆盖当前入口指向。

## 5. 非功能设计

| 维度 | 要求 | 实现策略 |
|---|---|---|
| 可用性 | 服务可观测 | `/health` + API 日志 + 邮件/海报任务状态 |

## 7. 风险与降级

| 风险 | 触发条件 | 降级方案 |
|---|---|---|
| 海报生成失败 | 模板渲染异常 | 结果页先展示文本报告，支持手动重试 |
| 邮件发送失败 | SMTP 不可用 | 记录 `email_jobs` 失败状态，不影响结果查看 |
| Redis 不可用 | 连接失败 | 在线人数接口返回 0 并记录 warning |
| 支付未开通 | 商户未配置 | 详细报告页只展示锁定预览和即将开放提示 |
| 后台查询量上升 | 数据增长 | 列表分页、索引和时间筛选优先实现 |
