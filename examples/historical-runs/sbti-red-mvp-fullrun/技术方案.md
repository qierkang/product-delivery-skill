# SBTI RED 免费开放版 + 详细报告解锁版技术方案

## 元信息

- 版本：`v1.2`
- 日期：`2026-04-11`
- 编写人：`xyqierkang@gmail（claude+codex）`
- 关联需求：`需求文档.md`
- 状态：`评审中`

## 1. 设计目标

- 面向微信 H5 传播场景设计，支持微信群、朋友圈、二维码海报等入口。
- 首版按 `500-1000` 同时在线目标设计，不追求超大分布式，但必须避免单点卡死。
- 用户从进入页面、答题、生成报告、保存海报、分享裂变、留邮箱、解锁详细报告，到运营后台分析，形成完整闭环。
- 代码、SQL、接口、事件均要求可读、可审计、可维护，避免后续交接靠口头说明。

## 1.1 已确认业务口径

- 题库采用 `31` 题标准版。
- 人格原型数量采用开源项目同等规模：
  - `25` 个常规人格
  - `2` 个特殊结果人格
  - 合计 `27` 个结果原型
- 免费报告只给人格结果、缩略说明和少量解释。
- 详细报告才拆分各维度含义和更完整的解释。
- 微信传播仅依赖 `sbti.red` H5 链接分享，不接公众号和微信 JS-SDK。
- 部署环境为国内天翼云 Linux 服务器，采用 Docker 方式部署。
- 邮件服务商采用 `163` 邮箱 SMTP/IMAP 能力，敏感凭证通过环境变量注入。
- 支付能力本轮只预留后端接口和数据结构，前台页面不展示。
- 海报视觉方向：`红黑 + 镜面 + 情绪值`
- 文案方向：`好玩有梗 + 半专业半娱乐`

## 2. 总体架构

### 2.1 应用分层

- 前台应用：`apps/web`
  - `React + Vite`
  - 只负责 H5 展示、交互、埋点上报、微信传播页体验
- 后台应用：`apps/manage`
  - `React + Vite`
  - 只负责数据看板、用户列表、报告分析、配置管理、异常任务查看
- API 服务：`apps/api`
  - `Node.js + Express`
  - 承载 session、题库、评分、报告、海报、邮件、订单、后台查询接口
- 任务处理层：
  - `Redis + BullMQ` 或同类任务队列
  - 异步处理海报生成、邮件发送、支付回调补偿、统计聚合
- 数据层：
  - `MySQL 8` 存业务主数据和埋点明细
  - `Redis 7` 存在线人数、短缓存、任务队列、接口限流、幂等锁
- 文件层：
  - 开发环境先用本地 `storage/`
  - 线上建议切 OSS/COS/S3，避免容器重启导致海报丢失
- 网关层：
  - `Nginx` 统一入口
  - `apps/web` 和 `apps/manage` 静态资源强缓存
  - `apps/api` 走反向代理和限流

### 2.3 部署口径

- 目标环境：天翼云 Linux 服务器
- 部署方式：Docker Compose 一次部署
- 推荐容器：
  - `nginx`
  - `web`
  - `manage`
  - `api`
  - `worker`
  - `mysql`
  - `redis`

### 2.2 高并发思路

- `1000` 在线不是高到必须上复杂微服务，但不能所有逻辑都同步做完。
- 首屏、题目、免费报告必须快返回；海报、邮件、明细聚合走异步。
- 静态页面和图片必须尽量走 CDN 或 Nginx 缓存，不能每次都打到 API。
- MySQL 只做主数据写入和分析查询，不承担在线状态统计。
- 在线人数、报告短缓存、任务状态查询优先走 Redis。

## 3. 容量与性能口径

| 项目 | 目标 |
|---|---|
| 同时在线 | `500-1000` |
| 报告生成接口 P95 | `< 800ms` |
| H5 首屏加载 | `< 2.5s`（4G 环境） |
| 海报生成 | `3-8s` 内完成，失败可重试 |
| 邮件发送 | 不阻塞主流程 |

### 3.1 性能关键策略

- `apps/web` 构建后静态托管，避免 SSR 增加复杂度。
- `/api/session/bootstrap`、`/api/tests/questions`、`/api/reports/:token` 做短缓存。
- `/api/tests/submit` 同步只做：
  - 写作答主表
  - 计算结果
  - 生成报告快照
  - 返回 `report_token`
- 海报生成、邮件发送改成异步任务。
- 报告页默认先展示文字结果，再轮询海报状态。
- 埋点异步入库，必要时支持批量写入。

## 4. 模块拆分

### 4.1 API 领域模块

- `session`：访客初始化、渠道识别、分享回流识别
- `quiz`：题库读取、作答提交、评分、结果匹配
- `report`：免费报告、详细报告、历史报告、重新测试
- `poster`：海报模板、二维码地址、异步渲染、重试
- `email`：报告邮件、失败重试、运营触达
- `payment`：下单预留、回调处理、解锁状态更新
- `analytics`：在线统计、漏斗统计、渠道分析
- `manage`：后台登录、看板、列表、配置、审计

## 4.3 接口设计

| 接口 | 方法 | 作用 | 说明 |
|---|---|---|---|
| `/api/session/bootstrap` | `POST` | 初始化会话 | 返回 `session_id`、随机昵称、系统配置 |
| `/api/tests/questions` | `GET` | 获取题库 | 返回当前题库版本与题目列表 |
| `/api/tests/submit` | `POST` | 提交答题 | 同步计算结果并返回 `report_token` |
| `/api/reports/:token` | `GET` | 获取免费报告 | 返回免费报告快照和海报状态 |
| `/api/reports/:token/detail` | `GET` | 获取详细报告 | 预留详细报告和解锁状态 |
| `/api/posters/:reportToken/regenerate` | `POST` | 重试生成海报 | 海报失败时触发重试 |
| `/api/emails/send-report` | `POST` | 发送报告邮件 | 异步写入邮件任务 |
| `/api/manage/dashboard` | `GET` | 后台首页统计 | 返回 UV、完测、留资、分享等指标 |
| `/api/manage/reports` | `GET` | 后台报告列表 | 支持按人格、留资、解锁状态筛选 |

### 4.2 后台功能模块

- Dashboard
  - UV、PV、在线人数、完测率、留资率、海报保存率、分享率、解锁点击率、付费率
- 渠道分析
  - `channel_code` 维度看 UV、完测、留资、分享回流、解锁
- 用户列表
  - `session_id`、昵称、邮箱、首次来源、最近活跃、是否解锁
- 报告列表
  - 人格结果、维度摘要、报告版本、海报、历史记录
- 邮件任务
  - 发送中、成功、失败、重试次数
- 异常任务
  - 海报失败、邮件失败、支付回调失败
- 系统配置
  - 题库版本、报告模板版本、海报模板版本、支付开关、邮件模板
- 审计日志
  - 谁查了什么、导出了什么、改了什么配置

## 5. 数据设计

### 5.1 设计原则

- 题库、人格、报告模板必须版本化。
- `report_token` 与 `share_token` 必须分离。
- 一次作答一条 `attempt`，逐题答案单独落表，方便看题目级流失。
- 埋点和业务主表分离，避免查询互相拖慢。
- 热数据和在线状态尽量不落 MySQL。

### 5.2 核心表域

| 表域 | 表名 | 作用 |
|---|---|---|
| 渠道域 | `acquisition_channels` | 渠道和二维码配置 |
| 访客域 | `visitor_sessions` | 会话、来源、分享进入关系 |
| 用户域 | `user_profiles` | 昵称、邮箱、运营触达 |
| 题库域 | `questionnaire_versions`、`question_items` | 31 题标准版版本管理 |
| 人格域 | `result_archetypes` | `27` 个结果原型和报告模板 |
| 作答域 | `quiz_attempts`、`quiz_answers` | 提交结果和逐题答案 |
| 报告域 | `report_snapshots` | 免费/详细报告快照 |
| 裂变域 | `share_visit_logs` | 分享回流统计 |
| 海报域 | `poster_assets` | 手机海报资源和重试状态 |
| 邮件域 | `email_jobs` | 报告邮件任务 |
| 支付域 | `payment_orders` | 解锁订单 |
| 分析域 | `page_events` | 全链路埋点 |
| 后台域 | `admin_accounts`、`admin_audit_logs`、`system_configs` | 运营后台和审计 |

## 6. 关键链路

### 6.1 入口与传播链路

1. 用户点击微信群/朋友圈/H5 链接进入。
2. 前端读取 URL 中的：
   - `channel_code`
   - `share_token`
   - `utm_*`
3. 调用 `/api/session/bootstrap`。
4. 服务端完成：
   - 生成 `session_id`
   - 记录入口来源
   - 识别是否来自分享回流
   - 返回随机昵称、当前题库版本、前台系统配置

### 6.2 答题与报告链路

1. 前端获取 `31` 题标准版题目。
2. 用户逐题作答，逐题埋点。
3. 提交后服务端同步完成：
   - 保存 `quiz_attempts`
   - 保存 `quiz_answers`
   - 计算 `15` 维分数
   - 匹配人格原型
   - 生成 `report_token`、`share_token`
   - 生成免费报告快照
4. 前端进入结果页。

### 6.3 海报链路

1. 报告生成成功后写入海报任务。
2. 任务处理器读取：
   - 人格名
   - 核心文案
   - 二维码目标地址
   - 海报模板版本
3. 渲染手机海报 PNG。
4. 回写 `poster_assets`。
5. 结果页轮询或 websocket 刷新海报状态。

### 6.4 邮件链路

1. 用户输入邮箱或报告页补填邮箱。
2. 服务端写 `email_jobs`。
3. 异步发送：
   - 免费版：摘要 + 回流链接
   - 已解锁版：详细报告入口
4. 失败不阻塞主流程，后台可重试。

### 6.4.1 邮件配置口径

- 服务商：`163`
- 发送协议：
  - `SMTP`
  - `IMAP`
- 仓库只保留以下环境变量名，不写真实密钥：
  - `MAIL_HOST`
  - `MAIL_PORT`
  - `MAIL_SECURE`
  - `MAIL_USER`
  - `MAIL_PASS`
  - `MAIL_FROM`

### 6.5 详细报告解锁链路

1. 用户点击“解锁详细解答”。
2. 本轮前台不展示支付和解锁入口，只保留后端、数据库、路由结构预留。
3. 若支付已开启：
   - 创建 `payment_orders`
   - 跳转微信 / 支付宝 H5
4. 支付回调成功后：
   - 更新订单状态
   - 更新 `report_snapshots.unlock_status`
   - 允许查看详细报告
   - 可触发详细报告邮件

## 10. 风险与降级

| 风险 | 影响 | 方案 |
|---|---|---|
| 微信瞬时流量上涨 | API 短时抖动 | 静态资源缓存、API 限流、异步任务削峰 |
| 海报生成慢 | 报告页卡顿 | 先出文字结果，海报异步补上 |
| 邮件服务波动 | 留资体验受损 | 队列重试，后台手动补发 |
| MySQL 写入上涨 | 分析慢 | 事件分表/归档、关键索引、管理端分页 |
| 题库或报告改版 | 历史结果错乱 | 题库、人格、报告模板版本化 |
| 分享链路统计不准 | 运营误判 | `share_token` 独立建模，回流单独记表 |
