# 00-需求总览（sbti-red-mvp）

## 1. 基础信息

- request-key：`sbti-red-mvp`
- 需求名称：`SBTI RED 免费开放版 + 详细报告解锁版`
- 需求类型：`新功能重定义`
- 当前阶段：`Requirement 评审中`
- 负责人：`qierkang`
- 日期：`2026-04-11`

## 2. 业务目标

1. 跑通“扫码进入 -> 可跳过邮箱 -> 随机昵称 -> 完成测试 -> 免费报告 -> 海报分享 -> 新用户回流”的裂变闭环。
2. 支持“邮箱留资 + 发送报告 + 详细报告 1.99 元解锁 + H5 支付预留”的后续变现链路。
3. 建立一套可运营的后台，能看访问、在线、留资、测试结果、付费预留数据。

## 3. 本轮设计结论

- 推荐产品策略：`免费基础报告 + 可跳过邮箱 + 付费详细报告 + 自动生成海报`
- 推荐工程策略：`一个仓库 + 公域 H5 前台 + 运营后台 + 单 API 服务`
- 推荐报告策略：`固定模板生成，不依赖实时 AI 推理`

## 4. 范围定义

- P0 范围：
  - H5 落地页、测试页、免费报告页
  - 首次进入邮箱弹框（可跳过）
  - 随机昵称生成、手动改名、重新随机
  - 免费报告缓存、重新测试、今日测试序号展示
  - 自动生成海报图片并落本地文件目录
  - 海报附带二维码，扫码可回到报告查看页
  - 访问日志、在线人数、核心埋点、后台统计
  - 留资邮箱记录与报告邮件发送
- P1 预留：
  - 详细报告页
  - 微信支付 / 支付宝 H5 下单与回调预留
  - 1.99 元解锁逻辑
- 本轮不做：
  - 真正的支付商户配置
  - AI 动态生成千人千面的长报告
  - 多语言版本

## 6. 当前风险

| 风险 | 影响 | 应对动作 |
|---|---|---|
| 功能一次性扩太多 | 开发周期拖长 | 按 P0 / P1 拆分，先落免费链路 |
| 邮件 / 海报 / 支付都走外部能力 | 联调成本高 | 邮件先支持 SMTP，支付先做适配接口 |
| 报告分享页没有情绪价值 | 分享率低 | 结果页强视觉、烟花特效、海报和排名提示一起做 |
| 运营后台和前台耦合过重 | 后续难扩展 | 同仓库分应用目录，前后台独立 UI |
